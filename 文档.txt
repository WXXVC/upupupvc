# Telegram 媒体下载与上传系统（Monolithic架构）

## 项目概述
开发一个单体的全栈应用（前后端不分离，同一项目），支持 Docker 部署和本地运行。用于下载各类媒体文件（视频流、图片、文件）并上传至 Telegram 频道。

## 技术栈要求
- **核心**: Python (FastAPI/Flask) 或 Node.js (Express + EJS/Pug模板)
- **前端**: 服务端渲染模板引擎 或 轻量SPA嵌入后端（单HTML+JS）
- **数据库**: SQLite（单文件，零配置）
- **任务队列**: 内存队列/线程池/异步任务（无需Redis）
- **部署**: 单容器Dockerfile，一键启动

## 核心功能模块

1. 认证与初始化
- 首次启动强制配置页面：
  - Telegram API ID、API Hash
  - 目标频道ID/用户名（如 @mychannel 或 -1001234567890）
  - 下载文件存储路径
- **关键交互**: 首次连接Telegram时，后端通过TDLib触发认证流程，前端轮询或WebSocket弹出验证码输入窗口（支持短信验证码和两步验证密码）
- 配置持久化到SQLite，支持后续在"参数设置"中修改

 2. 下载功能
2.1 下载类型支持
| 类型 | 处理方式 |
|------|----------|
| **m3u8/HLS视频流** | 参考 N_m3u8DL-CLI 逻辑：解析m3u8 → 多线程下载ts分片 → 合并为mp4 |
| **直链视频/音频** | HTTP多线程分块下载，支持断点续传（Range请求） |
| **图片** | 单线程/多线程下载，保留原始格式 |
| **通用文件** | 流式下载，大文件分块写入避免内存溢出 |
2.2 单个下载
- 输入框粘贴URL（支持m3u8、mp4、jpg、png、zip等任意直链）
- **智能识别**: 根据Content-Type或URL后缀判断文件类型
- 解析基本信息：
  - 视频：分辨率、时长、码率（ffprobe可选）、预估大小
  - 图片：尺寸、格式、大小
  - 文件：仅大小
- 可编辑：自定义文件名（含扩展名）、选择存储子目录
- 确认后创建下载任务
2.3 批量下载
- 文本域输入多行URL（每行一个，支持混合类型）
- 自动去重、校验URL有效性
- 批量创建独立任务，并行下载（可配置并发数）
2.4 下载任务管理
- **状态**: 等待中、解析中、下载中（实时速度、已下载/总大小、ETA）、已暂停、已完成、失败（含错误详情）
- **操作**: 
  - 暂停/继续（支持断点续传）
  - 取消（删除临时文件）
  - 重新下载
  - 打开所在文件夹
- **筛选**: 按状态、文件类型、日期范围
- **搜索**: 文件名、URL模糊匹配

3. 上传功能（基于 TDLib）
3.1 TDLib上传进度查询
**必须调研确认**: 查阅 TDLib 官方文档（https://core.telegram.org/tdlib/docs/），确认以下API能力：
- `uploadFile` 或 `sendMessage` 发送文件时，是否返回 `file` 对象的 `remote` 字段进度？
- 是否支持 `updateFile` 更新回调获取实时上传字节数？
- 如TDLib支持实时进度：前端展示百分比、上传速度、ETA
- 如TDLib仅支持状态查询：轮询获取状态（上传中/成功/失败），前端展示状态标签
3.2 上传任务管理
自动触发: 下载完成后自动加入上传队列（可开关）
手动上传: 选择本地文件直接上传
状态展示:
有进度：进度条 + 百分比 + 速度
无进度：状态徽章（🟡上传中 / 🟢成功 / 🔴失败）
操作: 取消上传（如TDLib支持）、重试失败任务
筛选/搜索: 同下载任务
4. 文件处理策略
4.1 后处理设置（下载完成后）
移动到指定文件夹: 强制验证路径可写，支持变量如 {date}/{type}
直接删除: 立即删除源文件
保留: 不动
上传成功后处理: 可独立设置（如上传成功后删除本地文件）
4.2 大文件分割（视频专用）
触发条件: 视频文件 > 2GB（Telegram单文件限制）
处理工具: FFmpeg -c copy 无损分割（不重新编码）
可配置项:
分割阈值（默认 2048MB，可调）
分割后文件命名规则（如 filename.part1.mp4）
上传逻辑: 分割完成后创建多个上传子任务，按顺序上传，描述中标注 part X/Y
5. 元数据与描述策略
文件名模式: {filename}（自动去除扩展名）
正则自定义: 用户输入正则，从URL或文件名提取内容作为描述
模板变量: {filename}, {date}, {size}, {type}, {part}（分割时）
实时预览: 输入模板后即时显示效果
6. 界面要求
架构: 单HTML应用或服务端渲染，通过REST API + WebSocket/SSE通信
布局:
顶部：全局状态、设置入口
左侧：下载任务列表（可筛选搜索）
右侧：上传任务列表（可筛选搜索）
底部/弹窗：参数设置、验证码输入
实时更新: WebSocket或Server-Sent Events推送任务进度
主题: 支持暗黑模式

实施方案（总体架构）
后端框架：FastAPI + Jinja2，服务端渲染主页面与配置页，JSON 接口 + WebSocket/SSE 事件流。入口为 main.py。
数据存储：单文件 SQLite，启动自动建库建表，模型与迁移管理集中在 models/.
任务执行：进程内 asyncio + 线程池混合，调度器维持全局与下载/上传独立并发信号量，支持优先级与批量入队。
Telegram 通道：唯一使用 TDLib，认证流程事件化推送给前端。
下载器：m3u8/HLS 解析 + 分片并发下载 + 合并；直链音视频 HTTP Range 分块断点；图片原格式；通用流式下载。
上传器：基于 TDLib uploadFile/sendMessage 与 updateFile 回调，缺失实时进度时轮询。
前端：单页服务端渲染主界面，左侧下载、右侧上传；顶部全局状态与磁盘空间；设置入口与认证弹窗；CSS 变量实现暗黑模式。
配置与守卫：首次启动检测配置表与完成标记，未完成只允许配置页与认证接口，其他页面强制跳转。
文件处理策略：下载后移动目录（含 {date}/{type} 变量）、删除/保留；上传成功后独立策略；大文件分割（默认 2GB），ffmpeg -c copy 无损分割，filename.part1.mp4 命名，上传自动生成子任务并带 part X/Y 描述。
日志：系统级与任务级日志入表与服务器日志文件，前端可拉取展示。
核心模块划分
配置管理（config/）：读写 SQLite 配置，校验完成标记。
Telegram 客户端（services/telegram/）：TDLib 连接、认证状态机、事件流。
下载器（services/downloader/）：m3u8、HTTP Range、图片、流式下载。
上传器（services/uploader/）：TDLib 上传封装与状态订阅。
任务调度（services/scheduler/）：队列、并发、状态机、重试策略。
进度推送（services/stream/）：WebSocket/SSE 广播。
文件处理（services/files/）：移动、删除、分割、模板变量解析。
模板渲染（templates/）：主页面与配置页。
数据模型（SQLite）
config：API ID、API Hash、目标频道、下载路径、并发数、分割阈值、处理策略、完成标记、TDLib 会话信息。
download_tasks：id、url、类型、状态、进度、速度、已下载、总大小、错误、保存路径、时间、重试次数、校验哈希。
upload_tasks：id、来源路径、目标频道、状态、进度、速度、已上传、总大小、错误、描述模板、分片序号/总分片、时间。
file_policies：策略配置。
logs：系统与任务级错误。
关键流程设计
首次配置流程：启动检测 config 表与完成标记 → 未完成强制进入配置页 → 保存配置 → 初始化 TDLib → 认证事件推送 → 成功后标记完成。
TDLib 认证：后端事件推送验证码/密码 → 前端弹窗输入 → 回传 → 状态更新持久化。
下载流程：校验/去重 → 创建任务 → 调度器取出 → 分类下载 → 进度推送 → 下载完成触发上传队列。
上传流程：检查分割阈值 → 分割生成子任务 → 依序上传 → 进度推送 → 执行上传后处理。
失败与重试：任务状态机 pending/running/paused/completed/failed/canceled，失败触发重试策略（指数退避/固定间隔）。
